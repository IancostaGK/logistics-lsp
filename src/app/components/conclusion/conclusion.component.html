Key Takeaways

1

LSP is about behavioural contracts, not just inheritance syntax

A subtype that compiles and inherits is not automatically LSP-compliant. The key question is: "Can I swap this subtype in without any caller knowing?" If the answer is no, the hierarchy is wrong.

2

A subclass that throws exceptions for inherited methods violates the contract

ElectricScooter.startEngine() throwing Error("No engine!") is the textbook violation. The parent promised a string ‚Äî the child delivers a crash. That is a broken contract, regardless of whether TypeScript allows it.

3

Over-applying LSP creates unnecessary abstractions ‚Äî use it when substitution is real

Giving every vehicle its own interface (ITruck, IMotorcycle, IElectricScooter) violates DRY and multiplies maintenance cost. Abstractions should emerge from genuine substitutability ‚Äî not be imposed on every type.

The Sweet Spot

"Apply LSP when objects are truly interchangeable in context."

When you can iterate a list of mixed types, call a shared method, and every type honours the contract ‚Äî you have achieved LSP. That is the goal.
