Liskov Substitution Principle

The L in SOLID ‚Äî a guide to safe inheritance

Formal Definition ‚Äî Barbara Liskov, 1987

"Objects of a subclass should be replaceable with objects of the superclass without breaking the application."

If S is a subtype of T, then objects of type T may be replaced with objects of type S without altering the correctness of the program.

When to Apply LSP

‚úÖ Apply LSP when‚Ä¶

  • ‚ñ∏ Subtypes are genuinely interchangeable in the same context (e.g. all vehicles that start an engine)
  • ‚ñ∏ You iterate a collection of mixed subtypes and call a shared method
  • ‚ñ∏ A method accepts a base type and must work with any subtype
  • ‚ñ∏ You want to extend behaviour without modifying existing code (Open/Closed)

‚ùå Don't force LSP when‚Ä¶

  • ‚ñ∏ Subtypes have fundamentally different contracts (electric vs combustion vehicles)
  • ‚ñ∏ You would need to throw exceptions or return dummy values to satisfy the parent's interface
  • ‚ñ∏ Creating a shared base requires duplicating fields across every subtype (‚Üí use Interface Segregation instead)
  • ‚ñ∏ There is only one concrete implementation ‚Äî abstraction adds no value
